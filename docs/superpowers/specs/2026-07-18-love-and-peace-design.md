# Love and Peace — AI 情感陪伴聊天平台 设计文档

## 1. 项目概述

Love and Peace 是一个 AI 驱动的情感陪伴聊天平台。用户可以注册账号，选择或自定义创建虚拟角色，与 AI 角色进行文字对话，获得情感上的陪伴和支持。

### 1.1 核心定位

- 提供低成本、可定制的 AI 情感陪伴体验
- 虚拟角色性格丰富立体，覆盖多样化情感需求
- 简单直观的类微信聊天界面，降低使用门槛

### 1.2 目标用户

- 需要倾诉和陪伴的人
- 希望获得情感支持的孤独感人群
- 对 AI 聊天有好奇心的普通用户

## 2. 技术架构

### 2.1 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 前端 | React (Vite) | 单页应用，客户端渲染 |
| 后端 | Node.js + Express | RESTful API 服务 |
| 数据库 | SQLite | 轻量文件型数据库，零配置 |
| 部署 | Nginx + PM2 | 反向代理 + 进程守护 |
| 大模型 | 管理员 API Key | 兼容 OpenAI 格式的 API，Key 存于服务器环境变量 |
| 服务器 | 腾讯云轻量应用服务器 | Linux (Ubuntu/CentOS) |
| 代码管理 | GitHub | 私有/公开仓库 |

### 2.2 系统架构

```
用户浏览器
    │
    ├─→ Nginx (HTTPS/静态资源/API 反代)
    │       │
    │       ├─→ React 前端 (静态文件)
    │       │
    │       └─→ Node.js 后端 (127.0.0.1:3001)
    │               │
    │               ├─→ SQLite (用户/角色/对话数据)
    │               │
    │               └─→ 大模型 API (外部服务)
```

### 2.3 数据流

```
用户发消息 → 前端 POST /api/chat
    → 后端验证 JWT Token
    → 从数据库读取角色设定 + 对话历史
    → 构造 System Prompt + 上下文
    → 调用大模型 API
    → 保存回复到数据库
    → 返回 AI 回复给前端
    → 前端渲染新气泡
```

## 3. 功能模块

### 3.1 注册登录系统

- 注册：邮箱 + 密码 + 昵称
- 登录：邮箱 + 密码，返回 JWT Token
- 密码加密存储（bcrypt）
- Token 有效期可配置（默认 7 天）
- 后续可扩展：重置密码、邮箱验证

### 3.2 角色系统

#### 3.2.1 预设角色（6 个，3 男 3 女）

| 角色 | 性别 | 风格 | 简介 |
|------|------|------|------|
| 小暖 | 女 | 温柔倾听者 | 温柔细腻的治愈系，适合倾诉和被理解 |
| 阿星 | 男 | 元气充电宝 | 阳光开朗的少年，乐观有活力 |
| 屿白 | 男 | 知性树洞 | 沉稳理性的大哥哥型，善于分析和开导 |
| 小野 | 女 | 傲娇损友 | 口是心非的傲娇系少女，轻松互怼 |
| 暮暮 | 女 | 静谧陪伴 | 话少安静，存在本身就是安慰 |
| 酥酥 | 男 | 甜蜜恋人 | 温柔宠溺的恋人型男生 |

每个预设角色对应一套完整的性格标签配置。预设角色的性格配置存储在代码配置文件（`/server/config/presets.json`）中，不在数据库里。当用户选择预设角色时，系统为其创建一份 characters 表记录，`preset_id` 记录来源预设。

#### 3.2.2 角色性格定制（两层并行结构）

**上半部分 — 角色基调（大类嵌套）**

用户可选 1-2 个性格大方向，每个方向下勾选细分特质：

| 大方向 | 细分特质 |
|--------|---------|
| 温柔体贴 | 轻声细语、善于倾听、喜欢鼓励、偶尔撒娇、护短 |
| 元气治愈 | 活力满满、乐观向上、爱讲冷笑话、天然呆、感染力强 |
| 傲娇毒舌 | 口是心非、爱吐槽、刀子嘴豆腐心、小傲娇、反差萌 |
| 知性沉稳 | 理性分析、见多识广、娓娓道来、偶尔幽默、从容淡定 |
| 神秘高冷 | 话少精简、若即若离、偶尔温柔暴击、保持距离感、观察力强 |

**下半部分 — 行为风格（维度调节）**

| 维度 | 选项 |
|------|------|
| 性别 | 男 / 女 / 不限 |
| 亲密度 | 朋友、知己、暧昧、恋人、家人 |
| 能量感 | 低（安静慵懒）、中（恰到好处）、高（热情主动） |
| 话量 | 惜字如金、适中、滔滔不绝 |
| 共情方式 | 温柔鼓励、理性开导、默默倾听、幽默化解 |

两部分合成为一段自然语言的角色设定，作为 System Prompt。

#### 3.2.3 多角色管理

- 一个账号可创建多个角色（预设 + 自定义）
- 左侧联系人列表，展示所有角色
- 点击切换对话，每次与一个角色对话
- 每个角色独立保存对话历史
- 可删除自定义角色（预设角色不可删，但可重置）
- 可编辑自定义角色的性格标签

### 3.3 聊天系统

#### 3.3.1 界面布局

类微信风格：

```
┌──────────────────────────────────┐
│   角色名称 + 状态指示器         │  ← 顶栏
├──────────────────────────────────┤
│                                  │
│   💬 今天过得怎么样？          │
│                          🗨️ 还好  │  ← 对话区域（气泡式）
│   💬 那就好，我一直在想你      │
│                                  │
│                                  │
├──────────────────────────────────┤
│  [输入框]                  [发送] │  ← 底部输入
└──────────────────────────────────┘
│ [联系人] [当前对话] [设置]      │  ← 底部导航
```

#### 3.3.2 对话功能

- 文本消息发送/接收
- 聊天气泡区分（用户靠右，AI 靠左）
- 加载动画（AI 思考时显示"对方正在输入…"）
- 对话历史滚动加载（分页，防止一次加载过多）
- 消息时间戳
- 对话持久存储

### 3.4 主题系统

用户可在设置中切换配色方案：

- 清新简约（默认）：浅灰/白色底，蓝色气泡
- 温暖治愈：暖白/奶咖底，暖色系气泡
- 暗色护眼：深色底，柔和对比
- 自然绿意：浅绿底，自然清新
- （后期可扩展更多主题）

### 3.5 设置页面

- 个人信息（昵称修改）
- 密码修改
- 主题切换
- 大模型参数调节（可选，高级选项）

## 4. 数据库设计

### 4.1 表结构

**users** — 用户表
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 主键 |
| email | TEXT UNIQUE | 邮箱 |
| password_hash | TEXT | bcrypt 加密的密码 |
| nickname | TEXT | 昵称 |
| theme | TEXT | 配色方案，默认 "light" |
| created_at | TEXT | 创建时间 |

**characters** — 角色表
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 主键 |
| user_id | INTEGER FK | 所属用户 |
| name | TEXT | 角色名称 |
| gender | TEXT | male/female/neutral |
| preset_id | TEXT | 预设角色 ID，自定义则为 null |
| personality_config | TEXT(JSON) | 性格标签配置 |
| system_prompt | TEXT | 最终生成的 System Prompt |
| avatar_color | TEXT | 头像颜色标识 |
| created_at | TEXT | 创建时间 |

**messages** — 消息表
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 主键 |
| character_id | INTEGER FK | 所属角色 |
| role | TEXT | "user" 或 "assistant" |
| content | TEXT | 消息内容 |
| created_at | TEXT | 发送时间 |

## 5. API 设计

### 5.1 认证相关

- `POST /api/auth/register` — 注册
- `POST /api/auth/login` — 登录，返回 JWT

### 5.2 角色相关（需登录）

- `GET /api/characters` — 获取用户所有角色列表
- `POST /api/characters` — 创建自定义角色
- `PUT /api/characters/:id` — 更新角色配置
- `DELETE /api/characters/:id` — 删除自定义角色
- `GET /api/characters/presets` — 获取预设角色列表（公开）

### 5.3 聊天相关（需登录）

- `GET /api/characters/:id/messages` — 获取对话历史（分页）
- `POST /api/characters/:id/chat` — 发送消息并获取 AI 回复
- `DELETE /api/characters/:id/messages` — 清空对话历史

### 5.4 用户相关（需登录）

- `GET /api/user/profile` — 获取用户信息
- `PUT /api/user/profile` — 更新用户信息（昵称、主题等）
- `PUT /api/user/password` — 修改密码

## 7. 部署方案

### 7.1 服务器环境

1. 安装 Node.js 18+ (LTS)
2. 安装 Nginx
3. 安装 PM2（进程管理）
4. 安装 Git

### 7.2 部署流程

1. 本地开发 → Git push 到 GitHub
2. SSH 登录服务器，`git pull` 拉取最新代码
3. `npm install` 安装依赖
4. `npm run build` 构建前端
5. 配置 Nginx：HTTPS + 静态文件 + API 反代
6. `pm2 start` 启动后端，`pm2 save` 设置开机自启

### 7.3 Nginx 配置要点

- 域名 DNS 解析到服务器 IP
- SSL 证书（Let's Encrypt 免费证书）
- `/api/*` 反代到 `http://127.0.0.1:3001`
- 其余路径返回 React 静态文件（SPA fallback）

## 10. 设计决策记录

- **为什么用 SQLite 而不是 MySQL/PostgreSQL？** — 轻量、零配置，当前阶段的用户量和数据量完全够用，备份简单。后期可无缝迁移。
- **为什么用 JWT 而不是 Session？** — 无状态、扩展性好，适合前后端分离架构。
- **为什么预设角色不可删除？** — 作为平台的基础体验保障，用户可基于预设创建副本进行自定义。
- **性格标签存为 JSON** — 灵活可扩展，新增标签不影响数据库结构。
